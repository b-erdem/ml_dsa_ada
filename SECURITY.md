# Security Policy & Threat Model

## Reporting Vulnerabilities

Report security issues privately via GitHub Security Advisories or email
`baris@erdem.dev`. Do not open public issues for vulnerabilities.

Disclosure SLA: acknowledgement within 7 days, fix or mitigation within
90 days for high-severity issues.

## Threat Model

`ml_dsa_ada` implements ML-DSA (Module-Lattice-Based Digital Signature
Algorithm), FIPS 204. The trust boundary is the public key: `Verify`
may be called on adversarially-chosen signatures and messages, on a
public key produced by an authentic `KeyGen`. The secret key is
trusted.

Coverage: type safety (range, overflow, index, division-by-zero),
loop termination, flow analysis (initialisation), `Always_Terminates`
on every public subprogram, and chained pre/post-conditions across
the rejection loop and the matrix/NTT pipeline.

The 8-layer forward NTT, the InvNTT-with-Mont-finalisation,
Power2Round, Decompose (both γ₂ variants, FIPS 204 Algorithm 36
direct form), MakeHint, UseHint, the rejection-sampling
`Poly_Challenge`/`Poly_Uniform_Eta`/etc., and the bounded
rejection-loop in Sign (`Max_Sign_Retries = 1000`, per-attempt
rejection probability ~2-4 %, failure probability < 2^-8000) are
all verified.

### What's verified empirically

#### Constant-time execution

**Static audit**: see [CT_AUDIT.md](CT_AUDIT.md). Critical
secret-dependent branches replaced with sign-mask arithmetic:
- `Reduce.CAddQ` — branchless via `Shift_Right_Arithmetic`.
- `Rounding.Decompose` — branchless centring + wraparound.
- `Rounding.Make_Hint` — unconditional Boolean ops (no short-circuit).
- `Poly_ChkNorm` — uses Ada's `abs` operator, which GNAT 14+ at -O2
  emits as the standard branchless `cltd; xor; sub` idiom.

**Disassembly verification**: `scripts/ct_disasm_check.sh`
disassembles `Reduce.CAddQ`, `Rounding.Decompose`,
`Rounding.Make_Hint`, and `Poly.Poly_ChkNorm` from the built
object files and asserts the only conditional jumps are (a)
overflow-detection (`jo`, statically dead for valid input) and
(b) forward jumps to `Constraint_Error` helper stubs (also
statically dead). Run after `alr build`. All four functions PASS
on x86-64.

**Welch t-test** (timing): `ct_harness/bin/ct_ml_dsa_sign 200` runs
Sign with two distinct keys and two random messages, alternates them
randomly, and computes Welch's t on the per-call wall-clock times.
On Apple Silicon (Rosetta x86_64), the test reports `|t| < 1` —
PASS at the dudect threshold of 4.5.

**Cachegrind** (D1/LLd misses): integrated into
`ct_harness/docker/run_cachegrind.sh`. Last-level data cache
miss delta between two distinct keys is 0.005 % (1 miss out of
20,015 — within toolchain noise). The 9 % D1 (L1) delta reflects
the FIPS 204 §3.4 rejection-loop iteration-count variance, not a
secret-leaking access pattern. See [CT_AUDIT.md](CT_AUDIT.md) for
the full table and class-A vs class-B miss counts at I, D1, LLd,
and LL levels.

#### FIPS 204 ACVP cross-validation

`tests/src/test_acvp_full.adb` is generated by
`tests/scripts/gen_acvp_full.py` from the official NIST ACVP-Server
JSON files. For the active parameter set it embeds **25 KeyGen + 15
SigGen (external/pure/deterministic) + 15 SigVer (external/pure)**
test vectors, runs each through the implementation, and compares
the byte-exact pk / sk / signature output (or accept/reject
decision) to the ACVP `expectedResults.json`.

All 55 vectors pass per parameter set, for all of ML-DSA-44/65/87
(165 byte-exact ACVP cross-validations total).

This is byte-exact cross-validation against the FIPS 204 reference,
not just self-consistency.

#### Fault injection

`Sign_With_Self_Verify` (alongside `Sign` in
[`ml_dsa-sign.ads`](src/ml_dsa-sign.ads)) is a defensive variant
of signing that re-verifies its own output against the supplied
public key. If verification fails — for instance, because a
voltage glitch, particle strike, or clock-glitching attack
corrupted an intermediate computation — the output buffer is
zeroed and `Ok` is set to `False`. Cost: ~1.5x of plain `Sign`.

This mitigates a narrow but well-documented class of fault attacks
on lattice signatures (loop-tampering and intermediate-value
corruption that produce a wrong-but-plausible signature). It does
not protect against attacks that subvert verification itself, nor
against attacks targeting key generation. For deployments where
fault injection is a concern, prefer this entry point over plain
`Sign`.

Other hardware side channels (power, electromagnetic emanation,
non-fault DPA) are out of scope. The implementation is
software-only.

#### Hedged vs deterministic signing

`Sign` accepts an explicit `Rnd : Byte_Array_32`. Pass 32 random bytes
from a CSPRNG for hedged mode (recommended; provides protection
against fault attacks on signing). Pass `[others => 0]` for the FIPS
204 deterministic mode. The library itself contains no entropy
source.

## Memory hygiene

[`ML_DSA.Wipe`](src/ml_dsa-wipe.ads) provides `Wipe_Byte_Array`,
`Wipe_Polynomial`, `Wipe_Poly_Vector_K/L`, and `Wipe_Poly_Matrix_KL`
procedures with `Inline => False` so the compiler cannot prove the
zeroising stores dead. `KeyGen` and `Sign` call these on every
secret-touching local before returning:
- KeyGen: `Seed_Buf`, `Hash_Out`, `Rho_Prime`, `Key`, `S1`, `S1_Hat`,
  `S2`, `T0`.
- Sign: `Key`, `Mu`, `Rho_PP`, `S1`, `S2`, `T0`, `Y`, `W`, `W0`,
  `CS2`, `CT0`, `Cp`, `Cp_Hat`.

If you build with whole-program LTO, add `-fno-builtin-memset` or
call `explicit_bzero(3)` instead, since LTO can see across the
`Inline => False` boundary.
